## 作ったもの

G検定の受験対策として、ブラウザだけで動く模擬試験Webアプリを作りました。

公開URL: https://gexam-practice.web.app

ただ、この記事で一番書きたいのはアプリの機能ではありません。

今回のポイントは、**スマホからChatGPTアプリでCodexにアクセスし、Codex経由で自宅の開発用PCを操作しながら作業した**ことです。

スマホでコードを直接書いたわけではありません。スマホから指示を出し、実際のファイル確認・編集・生成・整理は、自宅PC上のCodexに任せました。

## 流れを図にすると

今回の作業イメージはこうです。

![スマホからChatGPTアプリ、Codex、自宅開発PC、Firebase Hostingへつながる開発フロー](/images/gexam-practice-app/mobile-codex-home-pc-workflow-ja.svg)

流れとしては、次のようになります。

1. スマホでChatGPTアプリを開く
2. ChatGPTアプリからCodexに作業を依頼する
3. Codexが自宅開発用PC上のプロジェクトを確認する
4. Codexが必要なファイルを編集する
5. アプリや記事ドラフト、画像素材を作る
6. Firebase Hostingで公開できる形に整える

手元のスマホは、キーボード付きの開発端末ではなく、**開発指示を出すためのリモコン**のような役割です。

## 何が新鮮だったか

スマホだけで開発すると聞くと、普通はかなりつらい印象があります。

小さい画面でコードを読む。ファイルを探す。コマンドを打つ。Markdownを直す。画像を作る。パスを合わせる。

どれもスマホではやりにくいです。

でも今回のやり方では、スマホでやることはかなり減ります。

![スマホで直接コードを書く場合と、スマホを指示センターとして使う場合の比較](/images/gexam-practice-app/phone-only-vs-command-center-ja.svg)

```text
こういう記事にしたい
このファイルを見て
アプリ説明は少なめにして
スマホからCodex経由で自宅PCを操作している点を詳しくして
説明画像も作って
```

このように意図を伝えると、Codexがローカルのプロジェクトを確認し、実際の編集作業を進めます。

つまり、人間がスマホで細かく操作するのではなく、**スマホから目的を伝え、Codexが自宅PCで作業する**形になります。

![Codexが自宅PC上でファイル確認、記事編集、画像配置、チェックを進めるイメージ](/images/gexam-practice-app/codex-local-work-ja.svg)

## 今回のアプリについて

アプリ自体は、G検定向けの模擬試験Webアプリです。

機能は次のようなものです。

- 本番想定モード
- 短時間演習
- 分野別演習
- 弱点復習
- タイマー
- 即時採点と解説レビュー
- 間違えた問題のローカル保存

G検定は、用語暗記だけではなく、似た概念を区別して判断する力が必要です。

そこで、単語帳ではなく、本番に近いテンポで選択問題を解けるWebアプリとして作りました。

## プロジェクト構成

構成はかなりシンプルです。

```text
public/
  index.html
  styles.css
  app.js
  404.html
firebase.json
package.json
articles/
  qiita/
  zenn/
  assets/
```

Firebase Hostingには `public/` を配信させています。

```json
{
  "hosting": {
    "public": "public",
    "ignore": [
      "firebase.json",
      "**/.*",
      "**/node_modules/**"
    ],
    "cleanUrls": true,
    "trailingSlash": false
  }
}
```

小さな静的Webアプリなので、Codexが全体を把握しやすく、スマホからの自然言語指示とも相性がよかったです。

## スマホから頼んだ作業

今回、スマホからの指示で進めた作業は大きく次のようなものです。

- 既存のQiita/Zenn記事ドラフトを読む
- 記事の主題をアプリ紹介から開発体験に変更する
- スマホ、ChatGPT、Codex、自宅PC、Firebase Hostingの関係を説明する
- 説明用画像を生成する
- 生成画像を記事用のディレクトリに配置する
- Markdownから画像を参照する
- Qiita/Zennそれぞれの文体に合わせて本文を調整する

特に便利だったのは、記事本文と画像を同じ流れで扱えることです。

「この説明だと図がほしい」と思ったら、そのまま画像生成まで依頼できます。さらに、生成した画像を記事内のパスに合わせて配置するところまで任せられます。

## このやり方の強み

この開発スタイルの強みは、場所を選ばないことです。

外出中でも、スマホから思いついた改善を投げられます。自宅PCにはプロジェクト、Node.js、Firebase CLI、記事ファイルなど、いつもの開発環境があります。

スマホに全部を入れる必要はありません。

スマホは薄い操作面で、自宅PCが実行環境です。その間にCodexが入り、ファイル操作や編集を担当します。

この分担がかなり自然でした。

## 注意点

もちろん、便利だからといって完全放置でよいわけではありません。

少なくとも次は確認した方がよいです。

- Codexが編集したファイル
- 画像の保存場所
- Markdownの画像パス
- 公開URLや外部リンク
- デプロイ前のチェック結果
- 意図していないファイルが変わっていないか

特にローカルPCを操作できる状態では、対象範囲を明確にすることが大事です。

「このディレクトリのこのファイルを直して」と言えると、作業がかなり安定します。

## 著作権とセキュリティは大丈夫か

このやり方を発信するなら、便利さだけでなく、著作権とセキュリティの前提も書いておいた方がよいと思います。

今回のケースでは、記事本文、アプリ、説明図は自分で作成したものです。また、G検定の公式問題を転載しているわけではなく、シラバス範囲をもとにしたオリジナル問題として作っています。

そのため、気をつけるべき点は次のあたりです。

- 公式問題集や他サイトの問題文をそのまま使わない
- 他人の記事、図、スクリーンショットを無断転載しない
- ChatGPT、Codex、Firebaseなどの名称は説明上の固有名詞として使う
- 公式ロゴや公式認定のように見える表現は避ける
- 生成画像でも、第三者のロゴやキャラクターに寄せすぎない

セキュリティ面では、Codexが自宅PC上のファイルを読んだり、編集したり、コマンドを実行できる点に注意が必要です。

自分は次のような前提で使うのがよいと考えています。

- 作業ディレクトリを限定する
- APIキー、トークン、`.env`、認証情報を出さない
- 危険なコマンドやネットワークアクセスは承認制にする
- 変更差分を確認してから公開・デプロイする
- 記事に自宅PCのユーザー名、内部IP、秘密のパスを書かない

OpenAIのCodexドキュメントでも、Codexはサンドボックスと承認ポリシーを組み合わせて安全に使う前提で説明されています。ネットワークアクセスやサンドボックスを広げる設定はリスクが上がるため、慎重に扱うべきものです。

つまり、この記事で紹介しているのは「スマホから自宅PCを無制限に操作する方法」ではありません。

**対象範囲を限定し、必要な操作を確認しながら、スマホからCodexに開発を手伝ってもらう方法**として紹介するのが正確です。

## まとめ

今回作ったG検定アプリは、Firebase Hostingに載せたシンプルな静的Webアプリです。

でも本当におもしろかったのは、アプリそのものよりも、作り方でした。

**スマホ → ChatGPTアプリ → Codex → 自宅開発用PC → Firebase Hosting** という流れで、スマホだけを入口にして開発作業を進められました。

スマホでコードを書くのではなく、スマホで指示する。自宅PCの環境をCodexが使って作業する。

この形なら、スマホからでもかなり現実的に開発を進められます。

小さなWebアプリ、記事ドラフト、README整理、画像付きの技術メモあたりは、このスタイルとかなり相性がよさそうです。
